
# 项目目标

我们需要在多卡上做CKKS的加速计算。目前的硬件资源有8卡H100x4，8卡内使用NVLINK连接，机器间使用InfiniBand连接，机器间的通信和数据传输使用MPI和NCCL。

实现这些功能需要设计一个框架，用于统筹多机和多卡的计算。

当前项目主要是一个验证性的框架，用于验证类型设计，接口设计以及运行时设计是否合理。

# 建模

神经网络（如resnet20）等，经过下降后，生成一系列的CKKS指令，它们是SSA，与MLIR是两面一体的。
每一条指令的形态类似如下，以Add为例子：
%2 = %1 + %0,{devices=\[2,3\], compute_device=2, level= ...}
其中的deivces attribute表示%2的输出会同时分配到device 2和deive 3上。由于除了函数参数以外的每一个ssa value都是由op产生的（假设我们只有一个函数），因此我们可以将所有%x绑定在特定的一个或者几个device上。
而compute_device则是表示计算是在某个特定的device上进行的，compute_device需要从devices中选择。当devices的size!=1时，我们会将数据异步的广播到每张卡上。

由SSA我们可以构造出一个计算图，而由于每个算子都绑定了device和compute_device，我们可以在并行的执行这个计算图，并在执行的过程中mask掉当前device不需要计算的节点。

比如如下ssa:

```
1. %2 = %1+%0({devices=\[0, 1\], compute_device=0})
2. %3 = %2+%0({devices=\[1\], compute_device=1})
3. %4 = %3+%3({devices=\[1\], compute_device=1})
4. %5 = %2+%1({devices=\[1\], compute_device=1})
5. %6 = %5+%5({devices=\[0\], compute_device=0})
```

2个device同时执行计算图，当device1发现1.的op不是在当前设备计算时，屏蔽掉当前运算，进行下一个计算。然而，在执行%3的时候我们可能会发现%2没有准备好的问题，因此需要阻塞的等待%2的数据。
与此同时，device0在计算%2，当计算完成后，需要立即异步的广播到devices中标注的所有卡上。

以上是最朴素的例子，在实际使用过程中，有多种特殊的情况。比如，拓扑结构中单CPU不止控制一个device，比如需要做多机的并行计算。需要考虑通信，同步等问题。

具体而言，框架需要同时兼容三种计算模式：

1. 8卡GPU的无MPI模式：8卡间使用同一个CPU管理，因此不需要MPI，8卡间密文级并行

2. 8x4间的有MPI模式：8卡间使用同一个CPU管理，卡间使用多个MPI进行通信，不同机器运行同一个程序，通过mask去屏蔽不需要运行的op

3. 用于模拟的，在同一台机子的，使用明文的模拟程序。每一个节点只有一个计算资源，通过mask去屏蔽不需要运行的op

后续可能还需要考虑：
1. 多机间密文级别并行，单机8卡内rns级别并行

2. 可能考虑非GPU上运行，可能是多加速器，可能是多npu或者多cpu集群

3. 可能在测试时，只有1，4，6卡，不一定有八卡

> 当前建模是比较不完善的，尤其是计算模式并没有完善的建模，未来需要抽象出来一个更加完善的表示模式。
> 从ssa设计，到通信接口设计，再到runtime设计，可能都需要进行调整。



# 架构设计
为了应对以上建模，我们的架构设计如下：
1. 对于ssa，我们建模为ops结构体+ops序列。ops结构体忠实的表现每一个op的attributes，这样子的std::vector<ops>就忠实的表示ssa了。具体代码可以查看：./ssa_ops_def.hpp

2. 对于多卡间的的通信，我们通过./comm_interface.hpp 进行表示。注意需要考虑nvidia+cpu的异构架构下的主从特殊性（比如device间传送和host_to_device是不同的api）以及当前的接口设计能否让对接的人在接口内部通过条件判断来cover掉这些特殊性。另外就是，当前的接口设计能否和其他API对接？比如如果拷贝采用memcpy，那recive_async是不是就不必要（不必要的话采用空实现会有功能问题吗）；比如采用memcpy如何recive_fence；这一系列问题都需要考虑。

3. 对于计算图的执行，我们需要设计runtime，runtime主要负责几个内容：
  1. 顺序的从ssa指令序列中取op，调度在不同的卡上
  2. 如果存在多runtime并行（典型的如MPI下），runtime需要通过条件判断，判断当前op是否需要执行，如果不需要执行，则跳过。相当于进行了mask
  3. 如果存在多device通信，runtime需要在eval得到结果后，自动调用./comm_interface.hpp中的异步通信函数，发送或者广播数据到其他device上，同时在其他device上调用异步接收。当真正需要用到这些数据时，再调用fence去等待数据就绪。

runtime还需要应对单机内8卡和多机x8卡的调度情况。单机内八卡只有一个程序实例，多机则需要启动多个MPI程序。对于单机内的调度，为了能够兼容不同的卡数，不同的硬件（比如不再是H100，而是华为昇腾，寒武纪，google tpu），是否不应该让runtime去管理单机内的调度？而是通过一个抽象接口来去管理？这都是需要考虑的问题。
